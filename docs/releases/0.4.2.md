## SwanSong 0.4.2 beta — Make, Replay, Refine

SwanSong 0.4.2 brings the released SwanSong SDK 0.3.1 toolset into the signed
application payload and gives its newest game-development contracts native,
typed Studio controls. It also includes the latest shared visual-system and
screen-polish work from the 0.4 line.

Expected app identity: **SwanSong 0.4.2 (8)**.

## Before you install

SwanSong 0.4.2 requires macOS 14 or later. The official universal app supports
Apple silicon and Intel Macs. No games or original system firmware are
included; add only game, homebrew, and project material you own or are
authorized to use.

This beta must be selected explicitly from GitHub Releases. Stable-only clients
remain on their selected channel; beta-enabled clients can discover 0.4.2 after
its signed, notarized prerelease and beta appcast entry are published. Before a
manual install, verify `SwanSong-0.4.2-macOS-universal.zip` against
`SHA256SUMS.txt` and the matching release manifest from the same release.

## SwanSong SDK 0.3.1 is built in

Studio now embeds the complete tagged SDK 0.3.1 runtime, schemas, recipes,
Python package, documentation, and `swan` entry point. The app locks the exact
SDK commit and normalized content revision, creates a digest manifest for every
bundled file, and verifies that identity during build, packaging, launch, and
release checks.

Python 3.11+ and the Wonderful Toolchain remain honest local prerequisites.
Doctor reports their resolved versions and the SwanSong connection. **Choose
SDK…** remains an explicit framework-development override; **Use Bundled SDK**
returns to the verified 0.3.1 payload.

SDK 0.3.1 also fixes Doctor's local SwanSong probe for a valid persistent MCP
server. Doctor now reads one bounded response and cleans up its probe without
waiting for that server to exit, while retaining strict identity, protocol,
error, timeout, exit-code, and redaction checks.

## New Studio tools

### Visual Authoring

Assets can create, validate, report, and export the SDK's six public v1
authoring documents:

- tilemaps;
- sprites, animations, and hitboxes;
- color and monochrome palettes;
- collision regions and paths;
- scene flow; and
- audio timelines.

Exports retain the SDK's explicit integration status. Authoring previews and
handoffs are never described as gameplay or release evidence.

### Replay Timeline

Play can build a read-only frame timeline for the selected scenario, with
optional checkpoint, persisted-evidence, trace, and JSON-report bindings. The
timeline helps inspect exactly when inputs, checkpoints, and observations line
up; it does not replace a fresh SwanSong play run or inspected PNG/WAV evidence.

### Failing-plan Minimizer

Test can delta-reduce an exact-frame failure plan against a public SDK failure
predicate. Every candidate is evaluated through SwanSong, evaluation count is
bounded, and the SDK verifies from a fresh boot that the final minimized plan
still preserves the failure before it writes the result.

## Other changes in this build

- The first-party Homebrew Catalog uses its purpose-specific production trust
  key and continues to fail closed on signatures, publisher rights,
  provenance, immutable asset identity, byte count, and SHA-256.
- Studio's USB Hardware Lab keeps Doctor, update planning, digest-confirmed
  installation, and bounded physical control QA behind its exact pinned tool
  contract. Firmware writes still require a person at the Mac and remain absent
  from MCP automation.
- Shared colors, spacing, controls, and screen layouts have been refined and
  their reviewed snapshot coverage updated.

## Boundaries

- SwanSong is the only automated emulator and gameplay-validation backend.
- Studio invokes typed SDK commands; it does not contain a second manifest,
  asset, compiler, emulator, or release implementation.
- Local MCP remains off by default. Its Studio write allowlist is unchanged:
  it cannot select paths, create or directly edit projects, invoke the new
  author/replay/minimize controls, run Release, or execute arbitrary commands.
- No games or original system firmware are included.

## Best beta targets

- Create one project from each SDK recipe, then build, test, play, and inspect
  its release report using only the bundled SDK.
- Create and export each Visual Authoring document type, and confirm its
  integration status stays explicit rather than being presented as gameplay
  evidence.
- Replay a scenario with checkpoints, inspected evidence, and a trace; confirm
  the timeline matches the original exact-frame plan.
- Minimize a deliberately failing plan and confirm a fresh SwanSong boot still
  reproduces the same failure with the reduced plan.
- Run Doctor with both healthy and intentionally missing local prerequisites
  and confirm its results are bounded, actionable, and free of private paths.

## Known limits

- Python 3.11+, the Wonderful Toolchain, and the project-declared Wonderful
  packages remain external prerequisites; the bundled SDK does not silently
  install or replace them.
- Visual Authoring exports require normal game integration and validation.
  Replay timelines and minimized failure plans do not replace fresh gameplay
  runs or inspected PNG/WAV evidence.
- Commercial-title compatibility is still being validated. **Reached Video**
  is not an end-to-end verdict or an original-hardware accuracy claim.
- The first-party Homebrew Catalog is available through its signed production
  trust path. SwanSong Core downloads remain unavailable until that project's
  independent signed publication gates pass.

When reporting a beta issue, include the app build, macOS version, Mac
architecture, Studio workspace, SDK/Python/Wonderful identities, and sanitized
result. Do not attach ROMs, saves, private screenshots, audio, project paths,
or Translation Lab evidence.
